KrakenRelay Operator Guide

This guide explains how to operate, understand, and troubleshoot KrakenRelay.

Overview

KrakenRelay is a software-defined repeater controller. It performs the control functions of a traditional hardware repeater controller, but it does so in software using the host computer’s audio system, signal analysis, DSP processing, and PTT control.

In the current web build, the operator page exposes live status, device selection, repeater controls, identification settings, timeout timer controls, and PTT settings.

What KrakenRelay Controls

KrakenRelay is responsible for:

  • deciding when receiver audio should be repeated
  • applying software audio processing
  • keying and unkeying the transmitter
  • sending CW identification
  • generating courtesy and TOT tones
  • enforcing the timeout timer
  • reporting live status through the web UI

What KrakenRelay Does Not Control

KrakenRelay does not set or correct radio hardware alignment. It does not directly control RF gain, front-end sensitivity, PL/CTCSS decoding, transmitter deviation, RF power, or analog gain inside the sound interface. It also does not manage antennas, feedlines, site hardware, or external linking systems.

In other words, KrakenRelay controls the software audio path and transmitter keying, not the physical radio alignment.

Signal Path

The practical signal path is:

Receiver → audio interface → PortAudio/ALSA backend → receiver audio processing → software squelch detection → transmitter audio processing → PTT control → transmitter

This means KrakenRelay’s decisions are based on the audio that reaches the computer, not directly on RF strength. A strong on-air signal can still appear weak to KrakenRelay if the receiver or sound interface audio level is set too low.

Important
The audio meter in KrakenRelay is not an RF S-meter. It reflects digital audio level, not received signal strength at the antenna.

Understanding the Live Status Area

RX and TX Indicators

The RX light shows that received audio is above the software squelch threshold. The TX light shows that KrakenRelay is currently transmitting. The Running status pill shows whether the controller is stopped or running.

CPU, Load, and Uptime

The TEMP, LOAD, and UP pills report host-system health. Temperature shows the highest available CPU or NVMe temperature. Load shows the host’s 1-minute load average. Uptime shows host uptime, not repeater uptime.

These values are useful when diagnosing sluggish response, dropouts, or thermal issues on Raspberry Pi or other small systems.

PTT Status

The PTT status pill reports how KrakenRelay sees the transmit keying path. In VOX mode it reports VOX operation. In CM108 mode it checks whether the configured hidraw device exists and whether it is writable.

Messages such as Device Missing or Permission Denied point to Linux device-path or permission problems.

Audio Meter

The audio meter shows digital audio level, not RF signal strength. The value is displayed in dBFS:

  • 0 dBFS = full digital scale
  • Values near 0 dBFS indicate clipping risk
  • Normal speech should generally peak around -15 dBFS to -6 dBFS

A very important detail in this build: the meter display is effectively floored at about -60 dBFS, so a reading near -60 means “very quiet or below the displayed range,” not necessarily an exact measured noise floor.

TOT, Next ID, and Status Badges

The TOT readout and progress bar show how much transmit time has been used against the configured timeout limit. Next ID shows the countdown until the next scheduled CW identification. Sending ID indicates that an identification is being transmitted now.

If present, a TOT Lockout indication means the repeater has timed out and is temporarily preventing new transmissions.

Clipping and Limiting Badges

A Clipping indication means audio has reached digital full scale and distortion is likely. A Limiting indication means the limiter is actively reducing peaks to help prevent overload.

These indicators are best treated as warning signs that gain staging or limiter settings should be reviewed.

Maintenance and Restart Badges

The Restart badge indicates that the KrakenRelay controller is currently restarting. During this time the repeater will temporarily stop processing audio and the transmitter will not be keyed.

The Maintenance badge indicates that someone is actively working on the system. The repeater may still operate normally, but adjustments, restarts, or configuration changes may temporarily interrupt service.

These badges are informational and help operators understand why the repeater may briefly stop responding or behave differently during maintenance or system restarts.

Auto-Start Error Message

If Auto-Start is enabled but KrakenRelay cannot find the previously saved audio devices at startup, an error may appear near the top of the UI.

Device Change
This usually means a USB audio device changed numbering, was unplugged, or is no longer available.

Mumble Status

If your deployment includes a Mumble or linking status indicator, it generally shows whether the client service is running.

This does not always guarantee that a remote network connection is actually established.

Devices and Main Buttons

Input Device

The Input Device is the audio source from the receiver. This should be the sound interface connected to the receiver’s audio output, whether that is discriminator audio, speaker audio, or line audio depending on your installation.

If the wrong input device is selected, KrakenRelay will not detect signals correctly even if the receiver is hearing traffic.

Output Device

The Output Device is the audio path that feeds the transmitter. If the wrong output device is selected, KrakenRelay may key normally while sending no usable audio to the transmitter.

Start

The Start button launches the KrakenRelay controller using the selected audio devices. When the system starts successfully, status indicators begin updating and the repeater becomes operational.

Stop

The Stop button shuts down the repeater controller and forces the transmitter to unkey. Use this before maintenance, restarting the service, or changing critical settings.

Send CW ID

This button sends an immediate Morse code station identification. It is useful for testing CW settings such as pitch, speed, and volume, or for manually identifying the repeater when needed.

Test PTT

The Test PTT control is intended to key the transmitter briefly in order to verify that the PTT interface is functioning correctly.

This control is currently a placeholder for later implementation.

Configuration Controls

Lock / Unlock

Configuration settings are locked by default to prevent accidental changes during operation. Unlocking the configuration allows parameters to be edited.

Once changes are complete, it is recommended to lock the configuration again.

Save

The Save button writes configuration changes to disk so they persist after a restart. After saving, KrakenRelay briefly stops and then restarts the audio streams in order to apply the updated settings.

Operational Note
During this process the repeater may stop passing audio for a short period while the audio pipeline is reinitialized. This is normal behavior.

Auto-Start

When enabled, Auto-Start automatically launches the repeater controller when the KrakenRelay service starts. This allows the repeater to recover automatically after a system reboot.

Restart Required Badge

Some settings are safest applied after restarting the controller or service. If your build includes a restart-required indication, treat it as a reminder that certain changes may not take full effect until a restart has been performed.

Audio Settings

Sample Rate

  • Determines the digital audio sample rate used by the system.
  • Higher sample rates increase processing load but may improve fidelity.
  • This setting normally only needs to be configured during initial setup.

Chunk Size

  • Defines how many audio samples are processed at a time.
  • Smaller values reduce latency but increase CPU load.
  • Larger values increase stability but add a small amount of delay.
Operational Note
For deployments linking over Mumble, the chunk size should be set in multiples of 960. For most other deployments, the chunk size should be set in multiples of 1024.

High-Pass Filter

  • Removes low-frequency hum, rumble, and other unwanted low-end noise.
  • Improves speech clarity by reducing energy that does not help intelligibility.
Helpful Tip
Setting the HPF at 300Hz or lower will remove low-end noise without impacting speech. You can use the rx_spectrum and detect_hum scripts included in the software for help with setting your HPF and notch.

Limiter

  • Prevents transmit audio from exceeding safe digital levels.
  • Reduces peaks that could otherwise cause clipping or harsh distortion.
  • Helps maintain more consistent transmitted audio.

Compression

The compressor reduces dynamic range so that quieter speech becomes more intelligible while loud peaks are controlled. This can improve overall readability of repeater audio.

Compressor parameters are currently configured through the config.yaml file and are not exposed directly in the web interface. Changes to compression settings must therefore be made in the configuration file and applied by restarting or reloading the system.

Notch Filter

The notch filter removes narrowband interference such as steady hum or other constant tones that may be present in the receive audio.

Like compression, notch filter settings are configured through config.yaml and are not currently adjustable through the web interface.

Squelch Threshold

The squelch threshold determines when KrakenRelay considers a signal to be present. If set too low, the repeater may key from noise. If set too high, weak signals may not open the repeater.

Gain staging matters

KrakenRelay does not replace proper analog audio setup. Receiver audio that is too low can prevent reliable keying. Audio that is too high can cause clipping and poor transmitted quality.

A good starting point for many installations is approximately -40 dBFS with the default DSP configuration. This value may vary depending on receiver audio levels and the local noise environment.

Dual Output

Dual Output allows transmit audio to be sent to a second audio device. This can be used to feed an additional transmitter, a monitor system, or another downstream audio path.

Repeater Settings

Hang Time

Hang Time is how long the transmitter remains keyed after receive audio disappears. A shorter hang time keeps the repeater feeling tight and efficient. A longer hang time can make operation feel smoother, but it also occupies the channel longer.

Carrier Delay

Carrier Delay is primarily used when the transmitter is keyed using VOX rather than hardware PTT.

VOX Keying Behavior
When Carrier Delay is enabled, KrakenRelay briefly transmits a short noise burst to trigger the transmitter's VOX circuit. During this time, received audio is temporarily buffered. Once the transmitter has keyed and stabilized, the buffered audio is played back so the beginning of the transmission is not lost.

In hardware PTT configurations, this setting is usually less critical because the transmitter keys immediately. However, a small carrier delay can still be useful in some installations to ensure the transmitter is fully keyed before normal audio begins.

Anti-Kerchunk

Anti-Kerchunk helps prevent the repeater from responding to very brief accidental key-ups, squelch pops, or short noise bursts.

It does this by requiring a signal to remain present for a minimum amount of time before the repeater begins transmitting.

Courtesy Tone

The Courtesy Tone is a short two-tone "pip" that plays after a transmission ends, indicating that the repeater has returned to receive mode.

In the web interface, operators can:

  • Enable or disable the courtesy tone.
  • Adjust the courtesy tone volume relative to normal transmit audio.

The tone used by KrakenRelay is a two-tone pip sound. The exact tone frequencies can be modified through an advanced setting in the config.yaml file if customization is desired.

Identification

Callsign, Interval, CW Enable, Speed, Pitch, Volume

These settings control the station identification function. The operator can set the station callsign, the interval between IDs, whether CW ID is enabled, and the speed, pitch, and volume used for Morse code transmission.

CW identification should be configured to match the station’s regulatory requirements and operating practice.

Timeout Timer (TOT)

TOT Enable and Duration

The Time-Out Timer (TOT) protects the transmitter and repeater system from excessively long or stuck transmissions. When enabled, KrakenRelay measures continuous transmit time and automatically stops the transmission when the configured limit is reached.

Timeout Event
When the timeout occurs, KrakenRelay immediately unkeys the transmitter and stops passing audio from the receiver to the transmitter.

TOT Tone and Volume

When a timeout event occurs, KrakenRelay can generate a warning tone to indicate that the transmission was terminated by the Time-Out Timer.

The tone's pitch and volume can be adjusted so the timeout condition is clearly noticeable without being excessively loud.

Lockout and Lockout Time

After a timeout event, KrakenRelay may enter a lockout period. During this time the repeater will refuse new transmissions and will not pass audio to the transmitter.

This prevents the repeater from immediately re-keying if the timeout was caused by a stuck microphone, continuous input signal, or persistent noise condition.

The Lockout Time determines how long the repeater remains in this blocked state before normal operation resumes.

PTT Settings

PTT Mode

KrakenRelay typically supports VOX and CM108 keying modes. VOX depends on the transmitter’s own audio-operated keying behavior. CM108 mode uses a compatible USB audio device with GPIO exposed through a Linux hidraw node.

Device Path

In CM108 mode, the Device Path points to the Linux device node used for keying, usually something like /dev/hidraw0.

If the status shows Device Missing, the path may be wrong or the device may have changed numbering. If the status shows Permission Denied, Linux permissions or udev rules should be checked.

GPIO Pin

The GPIO Pin setting selects which CM108 GPIO line is used for PTT. This must match the way the interface is physically wired.

If PTT does not work even though the correct device is selected, verify the configured GPIO pin and the physical wiring.

If you are using a Digirig Lite or similar interface, you should select Pin 3.

Practical Operating Guidance

Setting Receive Audio

Receiver audio should be adjusted so that normal speech peaks fall roughly between -15 dBFS and -6 dBFS on the meter.

Audio that is too low may prevent the repeater from detecting signals reliably. Audio that is too high may cause distortion, clipping, or excessive limiter action.

Setup Tip
A helpful technique during setup is to use the Dual Output feature to send transmit audio to a second audio interface connected to a monitor speaker or headset. This allows operators to listen to the processed transmit audio directly while adjusting receiver levels and DSP settings.

Monitoring the transmit audio path can make it much easier to identify clipping, distortion, or excessive compression while calibrating the system.

Setting Squelch

Start by setting the squelch threshold slightly above the idle noise floor. If the repeater keys when no signal is present, increase the threshold slightly. If weak but usable signals fail to open the repeater, decrease the threshold slightly.

A good starting point for many installations is approximately -40 dBFS with the default DSP configuration. This value may vary depending on receiver audio levels and the local noise environment.

Additional advanced squelch behavior can be tuned in the config.yaml file. These parameters include:

  • Hysteresis — prevents rapid squelch chatter by requiring the signal level to fall several dB below the opening threshold before the repeater closes.
  • Hang Time — keeps the transmitter keyed briefly after a audio drops below the squelch threshold, which prevents the repeater from dropping between short pauses in speech.

Using Auto-Start

Auto-Start should be enabled only after the system has been tested and confirmed to start correctly with the selected devices.

It is a good practice to perform at least one successful manual start before relying on automatic startup after reboot.

Troubleshooting

Repeater will not key

Check that the correct input and output audio devices are selected, verify the squelch setting, and confirm that the PTT device is accessible to the system.

Repeater keys when no one is talking

This usually indicates a high noise floor, excessive receiver audio, or squelch set too low. Reduce receiver audio gain or increase the squelch threshold slightly.

Configuring the Carrier Validity variables in the config.yaml file can reduce transmissions that become stuck due to a high noise floor.

Audio is too quiet

Increase the receiver audio output level or the sound interface capture gain. Also verify that the correct input device is selected.

Audio is distorted

Reduce the receiver audio level or adjust the limiter settings. Distortion usually indicates that the input level is too high or that clipping is occurring somewhere in the chain.

Autostart fails after reboot

USB audio devices may have changed numbering. Re-select the correct devices and start the system again manually.

PTT Status says permission denied

The system does not have permission to access the PTT device. Check Linux device permissions, group membership, or udev rules.

System feels slow or frozen

Check system temperature and CPU load. If necessary, restart the KrakenRelay service after the underlying problem has been corrected.

Common operator checklist
  • Confirm the correct input and output devices are selected.
  • Check whether RX is opening when a signal is present.
  • Verify the audio meter shows sensible levels during speech.
  • Check PTT status for missing-device or permission problems.
  • Review squelch, timeout, and lockout settings.
